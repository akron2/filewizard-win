# File Wizard - Windows Edition Analysis

## Архитектура
Проект представляет собой self-hosted веб-приложение на FastAPI, которое предоставляет интерфейс для конвертации файлов, OCR и транскрипции аудио/видео.
- **Backend:** FastAPI, фоновые задачи (background jobs) реализованы через `huey` со SQLite. База данных для хранения задач — `sqlalchemy` со SQLite. WebSocket используется для обновления статусов задач на клиенте.
- **Frontend:** Jinja2 шаблоны, HTML/JS/CSS, отправляющие запросы к API и слушающие WebSocket.
- **Транскрипция:** ИСПОЛЬЗУЕТ `faster-whisper`.
- **Диалогизация (Speaker Diarization):** ИСПОЛЬЗУЕТ `pyannote.audio` (в файле `diarization.py`).

## Выявленные проблемы и пути решения

1. **Не работает диалогизация (HuggingFace Authentication & Terminal Blocking):**
   - **Причина:** В файле `diarization.py` при отсутствии токена HuggingFace вызывались функции, запускающие браузер и ожидающие CLI-ввода `input()`. В контексте веб-сервера это полностью блокировало фоновый поток Huey, замораживая задачи.
   - **Решение:** [✅ РЕШЕНО, PR #9] Из бэкенда удалены вызовы терминала. Теперь при ошибке авторизации генерируется исключение `TokenRequiredError`, перехватываемое задачей транскрипции, которая отправляет статус `hf_token_required` через WebSocket. На фронтенеде внедрено модальное окно, запрашивающее токен и сохраняющее его в `localStorage` для последующей инъекции в полезную нагрузку API запроса пайплайна.

2. **Не работает диалогизация (torchcodec DLL error / Issue #9):**
   - **Причина:** Библиотека `torchcodec` не могла загрузить FFmpeg DLL (`libtorchcodec_core8.dll`), потому что скрипт `setup_torchcodec.py` жестко копировал файлы от FFmpeg v6 (например, `avcodec-60.dll`). Так как `run.bat` скачивает актуальную версию FFmpeg (v7-v8), имена файлов изменились (например, `avcodec-62.dll`).
   - **Решение:** [✅ РЕШЕНО, Commit 405a3d0] Функция `copy_ffmpeg_dlls()` в `setup_torchcodec.py` исправлена. Теперь она динамически находит все `.dll` файлы в директории `bin` скачанного архива FFmpeg и копирует их в корневую папку модуля `torchcodec`. Таким образом гарантируется поддержка любой скаченной версии (включая FFmpeg 8).

2. **Не работает выбор модели транскрипции (отсутствие скачивания и запроса авторизации):**
   - **Причина:** Аналогично проблемам скачивания/авторизации, приложение не предоставляет UI для ввода токена во время запуска пайплайна транскрипции, или логика скачивания моделей Whisper/Diarization не обрабатывает случаи отсутствия локальной модели так, чтобы запросить авторизацию у пользователя в браузере. Выбор модели на фронтенде не учитывает статус "скачана/не скачана", а фоновый процесс падает с ошибкой без должного отображения в UI.
   - **Решение:** Добавить логику проверки наличия модели перед началом фоновой обработки. Если требуется токен для скачивания (например, pyannote), фронтенд должен сначала запросить его. Если модель скачивается, в WebSocket должен отправляться статус `downloading`, чтобы в UI отображался процесс загрузки, а не "зависание".

3. **Ошибка при запуске из-за torchcodec (FileNotFoundError DLL / Issue #10):**
   - **Причина:** В Python 3.8+ на Windows функция `ctypes.CDLL` не ищет зависимости загружаемых DLL в `PATH` из-за изменения политик безопасности. Из-за этого `torchcodec` не мог загрузить внутреннюю библиотеку `libtorchcodec_core8.dll`, так как система не находила скопированные FFmpeg DLL, и приложение выводило Traceback на этапе запуска. Кроме того, скрипт `setup_torchcodec.py` падал, так как не находил папку библиотеки внутри виртуального окружения (`site.getsitepackages()` на Windows возвращал глобальные пути).
   - **Решение:** [✅ РЕШЕНО, Issue #10] В `main.py` добавлен вызов `os.add_dll_directory` для папок среды FFmpeg и самого `torchcodec` перед его импортом. В `setup_torchcodec.py` логика поиска папки заменена на надежный `importlib.util.find_spec()`. Добавлен перехват `Exception` при импорте `torchcodec` в `main.py`.
